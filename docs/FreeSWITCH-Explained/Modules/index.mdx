# Modules

## 0. About

The functionality of FreeSWITCH can be expanded through the addition of modules that perform a particular task, whether that task is simple or complex. The modules are grouped into broad categories depending on

+ their **purpose** (e.g., application, codec, event handler)

+ whether they reside withing the FreeSWITCH source code (*"**in-source**" modules*) or not (*"**out-of-source**" modules*).

### 0.1 "in-source" vs. "out-of-source" modules

The *"**in-source**" modules* are the ones that reside within the [FreeSWITCH source code](https://github.com/signalwire/freeswitch) in the [<span class="fs-path">src/mod</span>][srcmod] directory.

The *"**out-of-source**" modules*' source code is located outside the FreeSWITCH source code and may be maintained by other than the FreeSWITCH developers.

## 1. Installing modules

:::warning

These general steps should work for most FreeSWITCH modules out of the box, but make sure to check the module's page that you would like to install for any extra steps or requirements.

:::

### 1.1 FreeSWITCH installed from source

1. Add / uncomment module's name in <span class="fs-path">modules.conf</span> in the FreeSWITCH source directory
2. Run `make`
3. Run `make install`
4. Add / uncomment module in <span class="fs-path">\<conf>/autoload_configs/[modules.conf.xml](#modules.conf.xml)</span> (see [Module configuration](#modules.conf.xml) below)
5. Load module in [<span class="fs-terminal-commands">fs_cli</span>][fs_cli] using `load <module_name>`
6. (optional) Check module's status in [<span class="fs-terminal-commands">fs_cli</span>][fs_cli] with `module_exists <module_name>`

#### 1.1.1 `modules.conf` and syntax

:::note[doc to-do]

Explain how `module.conf` works and why it is needed to add modules in there. I (toraritte) only have a rudimentary knowledge of Automake, Autoconf, and Autotools.

:::

[<span class="fs-path">modules.conf.in</span>](https://github.com/signalwire/freeswitch/blob/master/build/modules.conf.in) (which will become `modules.conf` when compiled from source) will always contain the most up-to-date list of FreeSWITCH modules maintained by FreeSWITCH developers.

Lines starting with `#` are ignored.

+ *"**In-source**" modules* are declared using their relative path within [<span class="fs-path">src/mod</span>][srcmod] in the FreeSWITCH source.

  Examples:
  + <span class="fs-path">applications/mod_commands</span>
  + <span class="fs-path">event_handlers/mod_event_socket</span>
  + <span class="fs-path">formats/mod_sndfile</span>

+ *"**Out-of-source**" modules* use the `<module_name>|<github_repo_url> -b <branch>` syntax.

  Examples:
  + `mod_freetdm|https://github.com/freeswitch/freetdm.git -b master`
  + `mod_kazoo|https://github.com/freeswitch/mod_kazoo.git -b master`

## 2. Important modules

:::info

FreeSWITCH can be compiled without modules, hence the use of "_important_" and not "_essential_". The only required dependencies are the C libraries provided by

+ [`spandsp`](https://github.com/freeswitch/spandsp),
+ [`sofia-sip`](https://github.com/freeswitch/sofia-sip), and
+ [`curl`](https://github.com/curl/curl).

Nevertheless, it is just easier to include the module wrapping them and adding extra functionality, without which it would probably be difficult (impossible?) to properly manage a FreeSWITCH instance.

:::

The list of "quality of admin-life" modules:

+ The aforementioned "wrapper" modules:
  + [<span class="fs-mod-names">mod_spandsp</span>](./mod_spandsp_6587021.mdx)
  + [<span class="fs-mod-names">mod_sofia</span>](./mod_sofia_1048707.mdx)
  + [<span class="fs-mod-names">mod_curl</span>](./mod_curl_3965033.mdx)

+ [<span class="fs-mod-names">mod_event_socket</span>](./mod_event_socket_1048924.mdx)

  [<span class="fs-terminal-commands">fs_cli</span>][fs_cli] needs this to connect to the running FreeSWITCH instance.

+ [<span class="fs-mod-names">mod_commands</span>](./mod_commands_1966741.mdx)

  Without this, there are only some very basic commands available on the [<span class="fs-terminal-commands">freeswitch</span>][freeswitch] console.

  :::note[doc to-do]

  Would such an instance even be able to do anything functional? Maybe as a "headless" installation?

  :::

+ [<span class="fs-mod-names">mod_console</span>](./mod_console_1048950.mdx)

  Makes [<span class="fs-terminal-commands">freeswitch</span>][freeswitch] console (and [<span class="fs-terminal-commands">fs_cli</span>][fs_cli]) friendlier. (Also controls the log levels.)

+ [<span class="fs-mod-names">mod_sndfile</span>](./mod_sndfile_32178375.mdx)

  From https://lists.freeswitch.org/pipermail/freeswitch-users/2010-November/064708.html

  > playback, IVR, record etc) requires it. It isn't necessary if you're
  > just routing calls between endpoints, but anything more complex
  > probably requires it.

  > -Steve \[Ayre]

## 3. Module configuration (<span class="fs-path">modules.conf.xml</span>) {#modules.conf.xml}

<span class="fs-path">modules.conf.xml</span> is a configuration entity that exists in the `configuration` section of [<span class="fs-path">freeswitch.xml</span>](../Introduction/Understanding-the-Configuration-Files.mdx). It denotes which modules to load, and in what order.

In order for a module to be loaded, it needs to have been compiled and installed to the proper location on the system. If you do not know how to compile/install a specific module, please read the [1. Installing modules](#1-installing-modules) section above.

:::warning
Modules are only loaded if the above conditions are met. Simply including a module's configuration in <span class="fs-path">freeswitch.xml</span> will not cause a module to load.
:::

[See the default <span class="fs-path">modules.conf.xml</span>](https://github.com/xrmx/freeswitch/blob/master/conf/autoload_configs/modules.conf.xml) that is used during most installations.

### 3.1 Setting a module to 'critical'

To prevent FreeSWITCH from starting at all if a specific module fails to load at startup, then use the `critical` attribute:

```xml
<load module="mod_really_important" critical="true"/>
```

If module `mod_really_important` fails to load, then FreeSWITCH will abort.

:::info
This is why the default [<span class="fs-path">modules.conf.xml</span>](https://github.com/xrmx/freeswitch/blob/master/conf/autoload_configs/modules.conf.xml) will always work: none of the enabled modules are marked as `critical`, so their failure to load (should it happen) will be ignored.
:::

### 3.2 Installing modules to a custom filesystem location

If the module <span class="fs-mod-names">mod_mymodule</span> is installed into <span class="fs-path">/opt/fs/mod_mymodule</span>, then it can be configured like so:

```xml
<load module="mod_mymodule" path="/opt/fs/mod_mymodule"/>
```
